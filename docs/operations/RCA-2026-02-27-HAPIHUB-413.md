## Summary

File uploads exceeding 1MB to `POST /storage/files` on HapiHub started returning **HTTP 413 Content Too Large** after the production gateway was migrated from Envoy Gateway to NGINX Gateway on Feb 26, 2026. NGINX Gateway enforces a default `client_max_body_size` of 1MB, whereas Envoy Gateway had no such default limit.

## Impact

- **All file uploads > 1MB** to HapiHub were rejected with 413 before reaching the application
- Affected the `POST /storage/files` endpoint used for patient documents, images, and attachments
- No data loss -- requests were rejected at the gateway layer, not the application layer

## Root Cause

### The Migration

Production HTTPRoutes were migrated from **Envoy Gateway** to **NGINX Gateway** in commit `35d0997`. The two gateways have different default behaviors for request body size limits:

| Gateway | Default Body Size Limit |
|---------|------------------------|
| Envoy Gateway | No limit (unlimited) |
| NGINX Gateway | `client_max_body_size 1m` (1MB) |

### The Gap

When the SnippetsPolicy was created for hapihub (`c6e069b`), only `proxy_read_timeout` was configured to fix WebSocket 502 errors. The `client_max_body_size` directive was not included because:

1. The 502 was the immediate symptom being addressed
2. The body size default difference between Envoy and NGINX was not identified during migration planning
3. No pre-migration checklist existed to verify feature parity between gateway implementations

### Request Flow (Where It Broke)

```
Client ──▶ NGINX Gateway ──▶ HapiHub API
              │
              ├─ TLS Termination
              ├─ client_max_body_size check  ◀── 413 REJECTED HERE (> 1MB)
              ├─ Proxy to HapiHub pod
              │
              │  (HapiHub never receives the request)
              │
              ▼ HapiHub Internal (if it had reached):
                 ├─ Koa Body Parser (jsonLimit: 50mb)
                 ├─ Multer Middleware (no file size limit)
                 └─ Storage Files Service (save to GCP/GridFS)
```

### Why Investigation Was Misdirected Initially

The 413 error coincided with recent rapid hapihub version changes (10.13.1 -> 10.15.2 -> 10.15.3 all on Feb 26), which led to initial suspicion that a code change in hapihub caused the issue. HapiHub was rolled back to the last known stable version (10.11.15) before the root cause was identified as an infrastructure-level configuration gap.

## Fix

Added `client_max_body_size 50m` to the NGINX SnippetsPolicy for hapihub, matching the application's `jsonLimit: '50mb'` body parser configuration.

### Files Changed

1. **`charts/hapihub/templates/snippetspolicy.yaml`** - Added `client_max_body_size` directive
2. **`charts/hapihub/values.yaml`** - Added `clientMaxBodySize: "50m"` default value
3. **`values/deployments/mycure-production.yaml`** - Added `clientMaxBodySize: "50m"` for production

### Before

```yaml
# snippetspolicy.yaml
snippets:
  - context: http.server.location
    value: |
      proxy_read_timeout 3600s;
```

### After

```yaml
# snippetspolicy.yaml
snippets:
  - context: http.server.location
    value: |
      proxy_read_timeout 3600s;
      client_max_body_size 50m;
```

## Lessons Learned

### What went well

- Deployment history analysis quickly identified the NGINX migration as the change window
- Rolling back hapihub to a known-good version was a reasonable first step, even though the issue was in infra
- The SnippetsPolicy mechanism was already in place, making the fix straightforward

### What went wrong

- **No migration checklist for Envoy -> NGINX**: The behavioral differences between gateways were not systematically cataloged before migration
- **No integration test for file uploads through the gateway**: E2E tests run at the application level and don't go through NGINX, so the 413 was invisible to tests
- **Multiple changes on the same day**: The NGINX migration, hapihub version bumps, and WebSocket 502 fixes all happened on Feb 26, making it harder to isolate the root cause
